# Security audit — 2026-05-07

Scope: every `route.ts` under `apps/medconcierge/src/app/api/` (149) and `apps/web/app/api/` (85), plus auth helpers, CSRF helper, webhook verifiers, workers, and `withTenant`. CSP/headers in `next.config.js` already exist and are not flagged again.

## CRITICAL findings (exploitable)

| File:line | Issue | Recommended fix |
|---|---|---|
| `apps/medconcierge/src/app/api/appointments/route.ts:9-52` | POST is fully unauthenticated AND trusts `tenantId` from request body (`createAppointmentSchema`, `validators/appointment.ts:9`). Any caller can `POST /api/appointments` with any tenant UUID and create real `appointments` + `patients` rows in any tenant. Only mitigation is 10/min IP rate limit. Used as the public booking API but without binding to a slug+secret it is open to enumeration. | Resolve `tenantId` from a server-trusted source: either the subdomain (`x-tenant-slug` header set by middleware) or a per-tenant booking token. Reject body `tenantId`. |
| `apps/web/app/api/admin/queue/route.ts:6-37` | GET is unauthenticated. Returns BullMQ counters AND payload (`tenantId`, `failedReason`) for the 10 most recent failed jobs. Also returns raw `err.message` on error. Leaks tenant identifiers and worker errors to the public internet. | Wrap in `requireRole(['super_admin'])`; never return `err.message` to clients. |
| `apps/medconcierge/src/app/api/internal/health/route.ts:7-11` and `apps/web/app/api/internal/health/route.ts:7-11` | Origin restriction is `forwarded.startsWith('127.0.0.1')`, applied only when `forwarded` is non-empty. (a) Anyone can spoof `X-Forwarded-For: 127.0.0.1, ...` — Nginx's default `proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for` *appends* rather than replaces, so the spoofed prefix wins. (b) When `forwarded` is missing entirely the check passes. Endpoint exposes DB latency, RSS/heap, uptime — useful for resource-exhaustion probes. | Either bind the route to localhost in Nginx and reject all forwarded requests, or require an internal shared secret header. Don't trust `X-Forwarded-For` for auth. |
| `packages/payments/src/billing-mercadopago.ts:72-94` | `verifyMPWebhook` HMAC is computed over `id;request-id;ts;` but `ts` is never compared to the current time. A captured webhook (e.g. an attacker who intercepts one MP delivery, or a recipient who replays a real one) is valid forever. Combined with `creditAddon` being idempotent only on `(processor, externalPaymentId)`, replay protection is just whatever Stripe-style header dedup the upstream provides — none here. | After signature verification, reject when `Math.abs(now - ts*1000) > 5*60*1000`. |

## HIGH severity findings (exploitable but limited blast radius)

| File:line | Issue | Recommended fix |
|---|---|---|
| `apps/web/lib/auth.ts:8-11` | `getSession()` calls `supabase.auth.getSession()` (cookie-trusted) instead of `getUser()` (server-validated). The medconcierge equivalent (`apps/medconcierge/src/lib/auth.ts:36-50`) correctly uses `getUser()`. On the web app, a forged/stale session cookie that decodes to a real JWT (e.g. after a server-side leak, or after Supabase rotates project keys) is accepted without re-validation. | Switch web's `getSession()` to use `getUser()` and return a session-shaped wrapper, mirroring medconcierge. |
| `apps/web/lib/auth.ts:24-39` and `apps/web/lib/auth.ts:65-79` | Auto-sync block: when a Supabase user ID has no row in `users`, look up by email and rewrite `users.id` to the Supabase UID, granting that Supabase identity the matched row's `tenantId` and role. This means any Supabase auth identity that signs up with the same email as an existing `users` row — including pending team invites whose row was created by `apps/web/app/api/dashboard/settings/team/route.ts:86-93` with a placeholder UUID — silently inherits the tenant. Email is verified by Supabase OTP, so practical exploit requires the attacker control the email. The risk is admins typo-inviting an attacker-controlled address into an `admin` seat. | Replace email-rewriting with an explicit `pending_invites` table; require the invitee to click a one-time invite link bound to a token before linking the auth ID. |
| `apps/medconcierge/src/app/api/onboarding/route.ts:77-167` | PATCH lacks `validateOrigin`. Same-origin policy is the only protection. Allows a CSRFed request to flip `skipAll: true`, marking onboarding as completed for the doctor's tenant. Cosmetic, but bypasses an audit hook. | Add `validateOrigin(request)` at the top of PATCH. |
| `apps/medconcierge/src/app/api/admin/tenants/[id]/route.ts:138-174` | PATCH has `requireRole(['super_admin'])` but no CSRF check. A super_admin browsing an attacker site could be CSRFed into mutating any tenant's `plan`, `provisioningStatus`, `isActive`, `config`. | Add `validateOrigin(request)`. |
| `apps/web/app/api/dashboard/settings/security/2fa/{disable,enable,verify}/route.ts` | None of the 2FA endpoints call `validateOrigin`. `disable` and `verify` require a valid TOTP code so CSRF can't directly disable 2FA without phishing the code — but `enable` only requires being authenticated, so a CSRFed POST/POST flow could trigger unexpected secret rotation. | Add `validateOrigin(request)` to all three. |
| `apps/web/app/api/dashboard/payments/create-link/route.ts:19-69` | POST creates Stripe/MP payment links without `validateOrigin`. A CSRFed admin's browser could be forced to mint payment links to attacker-controlled descriptions, polluting `payments` and showing fake links in the dashboard. | Add `validateOrigin`. |

The full list of web mutating routes missing CSRF (auth-gated, no HMAC, no API key) is reproduced from the scan and includes campaigns, conversations, integrations, follow-ups, funnel/move, invoices, payments, settings/{bot,channels,messages,subscription,team,team/[id]}, budgets, notifications. medconcierge equivalents that miss CSRF: `appointments/[id]/{cancel,reschedule,send-reminder}/route.ts`, `appointments/route.ts` (already CRITICAL above), `ai/{knowledge,playground}/route.ts`, `dashboard/me/push-token/route.ts`, `signup/route.ts`, `onboarding/{,apply-specialty}/route.ts`, `admin/tenants/[id]/route.ts`. Fixing the helper isn't enough — each route handler needs the call.

## MEDIUM severity (defense-in-depth gaps)

- `packages/db/index.ts:40-45` defines `withTenant()` but it is only invoked inside `packages/ai/tool-executors.ts` (AI tool calls) and `packages/ai/rag.ts`. Every API route, worker (`scripts/worker.ts`, `scripts/campaign-worker.ts`), and cron (`scripts/cron-*.ts`) executes against the connection without setting `app.tenant_id`. RLS policies (if any are wired up) cannot enforce isolation; a single forgotten `eq(table.tenantId, ...)` in any handler exposes every tenant. Same finding as `docs/archive/SECURITY-AUDIT-2026-04-20.md` HIGH-02 — still unaddressed for application routes.
- `apps/web/lib/csrf.ts:1-7` `validateOrigin` returns `true` when `origin` or `host` is missing ("Allow non-browser requests"). Browsers always set `Origin` on cross-site form POSTs, so this is mostly safe, but a deliberately stripped `Origin` header from a non-browser HTTP client (curl-style proxy, some service workers) bypasses CSRF. The medconcierge version (`apps/medconcierge/src/lib/csrf.ts`) correctly fail-closes. Align them.
- `apps/web/app/api/admin/health/route.ts:9-52` is unauthenticated and discloses `NODE_ENV`, DB/Redis status, and worker counts. Useful for fingerprinting; bind behind super_admin auth.
- `apps/medconcierge/src/app/api/auth/logout/route.ts` and `apps/web/app/api/auth/logout/route.ts` POST handlers do not call `validateOrigin`. A CSRFed logout is low impact (annoyance), but easy to fix. Web's logout additionally relies on `createSupabaseServer()` whose `setAll` is silently swallowed in some contexts; medconcierge's explicit `setAll` against the response is the right pattern — port it to web.
- `apps/medconcierge/src/app/api/webhooks/whatsapp/route.ts:399-503` `handleAppointmentKeyword` matches patient by phone using `LIKE '%' + normalized` (suffix match). Since the inbound is HMAC-verified and the keywords (`SI`/`NO`) act on the signed-in patient's record, exploitable only if Meta forwards a real message — but the suffix LIKE means a longer phone whose last 10 digits collide with another tenant's patient could match the wrong row. Switch to `=` after canonicalizing to last 10 digits.
- `apps/web/lib/auth.ts:7-11` calls `auth.getSession()` without a try/catch; a corrupted cookie throws and bubbles up as 500. Mirror medconcierge's defensive `try/catch` in `apps/medconcierge/src/lib/auth.ts:36-50`.
- Several integration routes return `err.message` to the client (`apps/medconcierge/src/app/api/dashboard/invoices/[id]/cancel/route.ts:58`, `.../stamp/route.ts:100`, `.../pdf/route.ts:56`, `.../integrations/{instagram,facturapi}/route.ts`, `apps/web/app/api/admin/queue/route.ts:35`). Facturapi/Instagram error strings can include API keys or internal trace fragments. Replace with a constant message; log full error server-side.

## LOW (best practice)

- Cookie domain helper `apps/web/lib/auth-cookie.ts:23` and the parallel medconcierge helper correctly set `.auctorum.com.mx`, so logout on `med.auctorum.com.mx` does clear the session for `dr-martinez.auctorum.com.mx` etc. Verified by walking the helper. No change needed.
- `apps/web/app/api/tracking/route.ts` uses 16-byte hex tokens (128 bits) generated in `apps/web/app/api/quotes/route.ts:123` via `crypto.randomBytes` — entropy is fine, no change.
- `apps/medconcierge/src/app/api/portal/[token]/files/[fileId]/route.ts` is a clean pattern: rate-limit on token, exact-match lookup, signed URL with 5-min TTL. Use as a template for any future bearer-token route.

## What's fine

- WhatsApp webhooks (`apps/medconcierge/src/app/api/webhooks/whatsapp/route.ts:32-48`, `apps/web/app/api/webhooks/whatsapp/route.ts`) use `crypto.timingSafeEqual` HMAC SHA-256 with the Meta app secret; the `wa/[slug]/webhook` route resolves the secret per bot instance.
- Stripe webhook (`apps/medconcierge/src/app/api/webhooks/stripe/route.ts:19-22` + `packages/payments/src/billing-stripe.ts:113-119`) uses `stripe.webhooks.constructEvent` which enforces a 5-minute timestamp tolerance by default.
- All `dashboard/*` GETs sampled (notifications, conversations/[id]/messages, patients/[id]/files/[fileId]) consistently chain `eq(table.tenantId, auth.tenant.id)` in the WHERE — no tenant-scoping omissions found in the GET surface.
- `apps/medconcierge/src/lib/auth.ts:41` correctly uses `supabase.auth.getUser()` (server-validated) rather than `getSession()` for medconcierge.
- v1 API surface (`apps/medconcierge/src/app/api/v1/*`) uniformly uses `authenticateApiKey` + permission check + per-tenant rate limit + `eq(.tenantId, auth.tenant.id)`. Solid.
- Public booking endpoints rate-limit by IP and validate Zod schemas; the issue with `appointments/route.ts` is the `tenantId` trust, not the schema or rate limit.
- 2FA TOTP (`apps/web/app/api/dashboard/settings/security/2fa/disable/route.ts`) decrypts secret via `decrypt()` with a legacy-fallback, verifies in a +/- 1 step window, and rate-limits 5/min — clean.
